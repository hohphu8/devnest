# DevNest 0.2.0

## Highlights

- Added an opt-in Laravel Octane Worker mode for FrankenPHP projects.
- FrankenPHP Classic remains the default and existing projects keep their current behavior.
- Worker-mode Laravel apps now run behind DevNest's normal local domain and tunnel origin flow, so `*.test` URLs and persistent domains do not need a separate manual origin.

## Added

- Project-level `FrankenPHP Mode`: Classic or Laravel Octane Worker.
- Octane preflight checks for Laravel, Composer metadata, `laravel/octane`, active FrankenPHP runtime, managed PHP extensions, and worker/admin ports.
- Per-project Octane worker state with managed ports, PID, status, last error, log path, worker count, and max request settings.
- Start, stop, restart, reload, status, settings, preflight, and log APIs for FrankenPHP Octane workers.
- Project Inspector controls for Octane settings, preflight, worker actions, and log preview.
- Browser mock coverage for the new FrankenPHP Octane API surface.

## Improved

- FrankenPHP worker startup on Windows now bypasses Laravel Octane's Artisan signal wrapper and runs the active `frankenphp` binary directly with Octane's bundled FrankenPHP Caddyfile.
- DevNest reuses the managed FrankenPHP PHP environment so project extensions load consistently with the active FrankenPHP runtime.
- Octane projects generate reverse-proxy Caddy config with forwarded host and HTTPS headers, fixing HTTPS asset generation behind DevNest's SSL ingress.
- DevNest can adopt an already-running project worker after app restart when the configured worker/admin ports are held by the same FrankenPHP process.
- Project start and Cloudflare tunnel origin startup now ensure the global FrankenPHP ingress and the project Octane worker are both running.
- Switching an Octane project back to Classic stops the project worker and regenerates the classic FrankenPHP config.

## Notes

- Laravel Octane Worker mode is Laravel-only in this release.
- To prepare a project, install Octane with `composer require laravel/octane`. DevNest uses Octane's vendor stubs directly; `php artisan octane:install --server=frankenphp` is not required for this local worker lane.
- Because Octane keeps Laravel booted in memory, restart the worker after code, config, `.env`, provider, or middleware changes.
- Apps that generate absolute HTTPS URLs behind the local SSL ingress may need Laravel trusted proxy configuration, then an Octane worker restart.
- Full Rust cargo check/test was intentionally not part of the fast verification path for this release workflow; `npm run verify:fast` is the primary verification command.
