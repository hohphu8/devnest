# DevNest 0.2.3

## Highlights

- Added portable FrankenPHP team profiles so a project can carry its mode, worker policy, PHP-family requirement, and extension requirements to another machine.
- Added FrankenPHP production export previews for Linux starter recipes.
- Made export/import safer by keeping secrets out of team profiles and showing compatibility warnings after import.

## Added

- Team project profile v2 for FrankenPHP projects.
- FrankenPHP metadata in team profiles, including Classic/Laravel Octane/Symfony/Custom mode, worker count, max requests, custom worker path, local SSL/domain intent, and local-only worker diagnostics.
- Runtime requirement metadata for shared FrankenPHP profiles, including embedded PHP family and required PHP extensions.
- Import-time compatibility warnings for missing active FrankenPHP runtime, PHP family mismatch, missing or disabled required extensions, unsupported worker mode, and local port portability.
- Production export preview and write flow for FrankenPHP projects from the Project Inspector.
- Generated production starter files:
  - `Caddyfile`
  - `devnest-frankenphp.service`
  - `Dockerfile`
  - `DEPLOYMENT.md`

## Improved

- Existing team profile v1 imports remain supported.
- Team profiles now export environment key names only, not tracked env values or `.env` secrets.
- Custom Worker production export preserves project-relative worker paths and avoids leaking Windows absolute paths.
- Laravel Octane production export warns when `public/frankenphp-worker.php` is missing.
- Symfony production export uses `public/index.php` and only emits the legacy `APP_RUNTIME` note when the project depends on `runtime/frankenphp-symfony`.
- Browser preview/mock mode now mirrors the new profile import result and production export APIs.
- Project import toasts now surface compatibility warning counts instead of treating every team import as a clean success.

## Notes

- Production export is a starter recipe, not a deployment tool. DevNest does not configure DNS, firewalls, production TLS certificates, SSH, CI/CD, or server users.
- Exported Caddyfiles default to HTTP starter configs with `auto_https off` so DevNest does not imply it has prepared production TLS. Review and adapt the Caddyfile before running it on a public server.
- Dockerfile output is a baseline FrankenPHP image and does not automatically install every native package needed by PHP extensions.
- Secrets are intentionally not exported. Share real production environment values through your own secret manager or deployment process.
