# DevNest 0.2.2

## Highlights

- Expanded FrankenPHP Worker mode beyond Laravel Octane with Symfony Worker and advanced Custom PHP Worker lanes.
- Kept the same project-domain routing model across Classic, Laravel Octane, Symfony Worker, and Custom Worker modes.
- Hardened worker startup, routing, and migration behavior for existing DevNest app data.

## Added

- Symfony Worker mode for Symfony projects using FrankenPHP Runtime support.
- Advanced Custom Worker mode for FrankenPHP projects with an explicit project-relative `.php` worker file.
- Generic FrankenPHP worker storage that carries mode, worker/admin ports, worker count, max requests, PID, status, logs, and custom worker path metadata.
- Symfony project detection from Composer packages and common Symfony files such as `bin/console`, `public/index.php`, and `config/bundles.php`.
- Mode-aware worker preflight checks for Symfony Runtime readiness, custom worker file validity, active FrankenPHP runtime, PHP family compatibility, and port availability.
- Runtime UI controls for starting, stopping, restarting, reloading, checking health, and reading logs for all worker modes.

## Improved

- Existing Laravel Octane worker settings are migrated into the new generic worker table without changing Octane behavior.
- FrankenPHP ingress now routes every non-classic mode through the per-project worker port while Classic remains direct `php_server`.
- Symfony and Custom workers now use DevNest-generated per-project Caddyfiles and the active managed FrankenPHP runtime.
- Worker-mode project start now syncs managed ingress configs and starts or restarts the global FrankenPHP ingress so new worker domains load immediately.
- Symfony native `symfony/runtime` support is detected without forcing the legacy `runtime/frankenphp-symfony` `APP_RUNTIME` override.
- Symfony/Custom worker Caddyfiles bind internally to `127.0.0.1` while accepting the preserved project `Host` from DevNest ingress.
- Quick tunnels and persistent domains now resolve worker-mode projects to the correct internal worker origin.
- Stopping global FrankenPHP now handles all managed FrankenPHP workers, not only Laravel Octane workers.
- Runtime Start/Restart now saves pending worker settings first, so Custom Worker file changes do not require a separate Save Settings click.
- Custom Worker errors now surface blocking preflight messages, such as a missing or invalid worker file, instead of only reporting a raw process exit.
- SQLite migration ordering was hardened so older DevNest app-data databases can rebuild project metadata safely.

## Notes

- Classic mode remains the default and is unchanged.
- Laravel Octane Worker mode remains Laravel-only.
- Symfony Worker and Custom Worker are opt-in advanced modes.
- DevNest detects missing Symfony Runtime support and shows Composer commands; it does not install Composer packages automatically.
- Custom worker paths are stored project-relative and must stay inside the project directory.
- WordPress worker mode, production export recipes, team profile portability, and automated benchmark comparisons remain deferred.
