# DevNest 0.2.8

## Highlights

- Added Redis as a managed optional tool and Restic-backed deduplicated storage for new Database Time Machine snapshots.
- Fixed a Windows localhost latency issue that could make some PHP projects take about 2 seconds longer to load.

## Added

- Redis is now available through the optional tools installer and can be started from Services.
- Restic is now available as a managed optional tool for deduplicated Database Time Machine snapshots.
- New Time Machine snapshots can use Restic dedup when Restic is installed, while legacy SQL snapshots remain restorable.

## Improved

- Redis service launch now falls back to the system Redis binary if the managed Windows Redis package exits immediately.
- Managed Redis now listens on both `127.0.0.1` and `::1`, so projects using `REDIS_HOST=localhost` avoid IPv6 fallback delays.
- Managed MariaDB now binds to `127.0.0.1,::1`, so projects using `DB_HOST=localhost` avoid repeated 2-second connection delays on Windows. Non-MariaDB MySQL runtimes keep the conservative IPv4 loopback bind.
- FrankenPHP worker admin and metrics calls now use `127.0.0.1` internally to avoid the same localhost resolver fallback class.
- Optional tool version display and install status labels are cleaner for Redis packages.

## Notes

- If a project still feels slow after restarting Redis and MySQL, check whether it is using another network dependency or an external URL during request boot.
- Projects can still use `127.0.0.1` explicitly in `.env`; DevNest now also handles `localhost` better for managed Redis and MariaDB local services.
